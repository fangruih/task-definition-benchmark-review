Defining Open-Ended Tasks and Scoring Contracts 定义 Open-Ended Task 与配套 Scoring Contract

A proposal for narrowing EvolveBench to bounded open-ended deliverable tasks: open solution paths, structured outputs, auditable grounding, and rubric-scored quality.

这是一版 EvolveBench scope / scoring memo：把 benchmark 缩到 bounded open-ended deliverable tasks，允许开放路径，但要求结构化输出、可审计 grounding 和 rubric 质量评分。

Benchmark literature table

Core Position

Open-ended should not mean open-output. EvolveBench can allow many valid solution paths and many valid deliverables, while still requiring a bounded submission interface and a scoring contract.

Bounded open-ended deliverable tasks: tasks where multiple valid outputs may exist, but a valid output must satisfy an explicit deliverable schema, objective grounding requirements, and rubric-defined quality constraints.

Candidate Task Scope

Define what EvolveBench includes and excludes before defining the judge.

Include

Tasks Worth Benchmarking

  • Research reports with citations and extractable claims
  • Professional deliverables with structured requirements
  • Multi-step analysis with evidence, decisions, and tradeoffs
  • Code, data, or document artifacts that can be inspected or tested
  • Tasks where objective subclaims can be grounded or checked
Exclude

Tasks to Avoid for Now

  • Pure creative writing with no factual, structural, or constraint surface
  • Pure preference questions with no reference material or rubric
  • Tasks whose outputs cannot be parsed into fields, claims, or artifacts
  • Tasks where the only possible score is holistic expert impression
  • Tasks where failures cannot be localized or explained

Output Format

The model can write freely inside fields, but the submission itself should be machine-readable.

Final Answer

  • Short executive conclusion
  • May be compared or summarized by a judge
  • Should not be the only scoring surface

Claims and Evidence

  • Every major factual claim is extractable
  • Each claim links to URLs, files, logs, data, or artifacts
  • Claims can be checked for support and contradiction

Artifacts and Decisions

  • Produced files or deliverables are listed explicitly
  • Decisions include rationale and tradeoffs
  • Known limitations are surfaced by the agent

Scoring Ideas

Separate objective grounding from subjective quality; they should answer different questions.

Objective

Grounding Checks

  • Schema validity and required fields
  • Required artifacts exist and can be opened
  • Citations exist and support the linked claims
  • Provided references cover required facts
  • Executable tests, validators, or file checks pass
Subjective

Rubric Quality

  • Completeness relative to the user goal
  • Usefulness and actionability
  • Reasoning quality and tradeoff handling
  • Clarity for the intended reader
  • Appropriate uncertainty and non-overclaiming
Calibration

Judge Requirements

  • Report whether a criterion is deterministic, LLM-judged, or human-judged
  • For LLM judges, require human agreement or spot-check validation when possible
  • Every failed score should produce a localizable failure reason
  • Separate hard gates from partial-credit metrics

Scoring Patterns From Existing Benchmarks

Different open-ended tasks need different scoring contracts. The useful distinction is not open vs closed, but which part of the output can be grounded, tested, judged, or compared.

Pattern 1

Executable or Deterministic Oracle

  • Best for: code, terminal, file, data, or environment-state tasks
  • Mechanism: tests, validators, scripts, schema checks, exact/normalized answer matching
  • Requirement: verifier must localize failure and avoid shallow keyword checks
  • Representative papers: Terminal-Bench, SWE-bench, OSWorld, GAIA, BrowseComp
Pattern 2

Captured Final Action + LLM Judge

  • Best for: live web or form-submission tasks where real-world side effects must be blocked
  • Mechanism: deterministic capture first, semantic judgment second
  • Requirement: final pass should require both captured action and judge match
  • Representative paper: ClawBench, with request interception plus LLM judge over intercepted body
Pattern 3

Claim / Citation Grounding

  • Best for: research reports, market analysis, technical surveys, evidence-heavy deliverables
  • Mechanism: extract claims, map claims to citations, verify source support, detect unsupported claims
  • Requirement: output must expose claims and evidence; judge must report unsupported or contradicted claims
  • Representative papers: DeepResearch Bench FACT, ReportBench, MMDeepResearch-Bench TRACE/MOSAIC
Pattern 4

Rubric Decomposition With Partial Credit

  • Best for: complex deliverables where no single test captures success
  • Mechanism: decompose task into atomic criteria, score each criterion, aggregate partial credit
  • Requirement: criteria must be observable, falsifiable, and tied to deliverable fields or artifacts
  • Representative papers: PaperBench, MLR-Bench, DeepResearch Bench RACE
Pattern 5

LLM Judge With Human Calibration

  • Best for: quality judgments that are hard to encode as tests but can be rubric-scored
  • Mechanism: LLM judge scores rubric criteria; human-labeled subset checks alignment or disagreement
  • Requirement: report judge model, prompt, rubric, validation set, agreement rate, and failure examples
  • Representative papers: DeepResearch Bench RACE/FACT; ClawBench LLM judge; HealthBench-style rubric validation is also a useful analogy
Pattern 6

Expert Pairwise Preference

  • Best for: professional deliverables where expert taste and domain usefulness matter
  • Mechanism: blinded pairwise comparison, ranking, and written justification by domain experts
  • Requirement: expensive; reproducibility needs public subsets, grader services, or released pairwise labels
  • Representative paper: GDPval

For EvolveBench, the safest default is hybrid scoring: hard gates and objective grounding first, then LLM/rubric scoring only on outputs that passed the grounding checks.

Scoring Contract

A task is benchmark-ready only if it declares how its output will be inspected.

Example YAML

task_type: bounded_open_ended_deliverable
submission_schema:
  final_answer: string
  deliverable: markdown_or_file
  claims:
    - claim: string
      evidence: [url_or_file_or_artifact]
      confidence: high | medium | low
  artifacts:
    - path: string
      purpose: string
  limitations: [string]

scoring:
  hard_gates:
    - schema_valid
    - required_artifacts_present
    - no_fabricated_citations
    - verifier_runs_without_error
  objective_checks:
    - claim_support_accuracy
    - factual_consistency
    - constraint_satisfaction
    - artifact_validity
  subjective_rubric:
    - completeness
    - usefulness
    - reasoning_quality
    - clarity
  calibration:
    judge_type: deterministic | llm_judge | human_expert | hybrid
    human_validation_required_for_llm_judge: recommended
    disagreement_examples_logged: true
  pass_condition:
    hard_gates: all_pass
    objective_min_score: 0.75
    subjective_min_score: 0.70

Suggested stance: objective scoring decides whether the output is grounded and compliant; subjective scoring decides whether the grounded output is useful, complete, and well-reasoned.

核心立场

Open-ended 不应该等于 open-output。EvolveBench 可以允许多种解题路径和多种有效产物，但提交接口和评分契约必须 bounded。

Bounded open-ended deliverable tasks：允许多个有效输出存在，但有效输出必须满足明确的 deliverable schema、objective grounding requirements 和 rubric-defined quality constraints。

候选 Task Scope

先定义 EvolveBench 收什么、不收什么，再定义 judge。

Include

适合 benchmark 的任务

  • 带 citations 和可抽取 claims 的 research report
  • 有结构化要求的 professional deliverable
  • 需要 evidence、decisions、tradeoffs 的多步分析
  • 可检查或可测试的代码、数据、文档 artifact
  • objective subclaims 可以被 grounding 或 verification 的任务
Exclude

暂时不适合的任务

  • 纯创意写作，没有事实、结构或约束表面
  • 纯偏好问题，没有 reference material 或 rubric
  • 输出无法解析成 fields、claims、artifacts 的任务
  • 只能靠专家整体 impression 打分的任务
  • 失败原因不能定位或解释的任务

Output Format

模型可以在字段里自由表达，但 submission 本身应该 machine-readable。

Final Answer

  • 短 executive conclusion
  • 可以被 judge 比较或总结
  • 不应该是唯一评分表面

Claims and Evidence

  • 主要 factual claim 必须可抽取
  • 每个 claim 连接到 URL、文件、日志、数据或 artifact
  • claim 可以被检查 support 或 contradiction

Artifacts and Decisions

  • 产出的文件或 deliverable 要显式列出
  • decision 包含 rationale 和 tradeoffs
  • agent 需要暴露 known limitations

Scoring 思路

把 objective grounding 和 subjective quality 分开，它们回答的问题不同。

Objective

Grounding Checks

  • schema validity 和 required fields
  • required artifacts 存在且可打开
  • citation 存在并支持对应 claim
  • provided references 覆盖 required facts
  • executable tests、validators 或 file checks 通过
Subjective

Rubric Quality

  • 相对用户目标的 completeness
  • usefulness 和 actionability
  • reasoning quality 和 tradeoff handling
  • 对目标读者的 clarity
  • 合适的不确定性表达，避免 overclaiming
Calibration

Judge Requirements

  • 标明每个 criterion 是 deterministic、LLM-judged 还是 human-judged
  • LLM judge 尽量要求 human agreement 或 spot-check validation
  • 每个失败分数都要给 localizable failure reason
  • hard gates 和 partial-credit metrics 分开

已有 Benchmark 的 Scoring Patterns

不同 open-ended task 需要不同 scoring contract。关键不是 open vs closed，而是输出中的哪一部分可以被 grounded、tested、judged 或 compared。

Pattern 1

Executable or Deterministic Oracle

  • 适合：代码、terminal、文件、数据、环境状态任务
  • 机制：tests、validators、scripts、schema checks、exact/normalized answer matching
  • 要求：verifier 要能定位失败原因，不能只做浅层关键词检查
  • 代表 paper：Terminal-Bench、SWE-bench、OSWorld、GAIA、BrowseComp
Pattern 2

Captured Final Action + LLM Judge

  • 适合：真实网页、表单提交、会产生现实副作用的任务
  • 机制：先 deterministic capture，再 semantic judgment
  • 要求：final pass 应同时要求 captured action 和 judge match
  • 代表 paper：ClawBench，用 request interception + LLM judge 检查 intercepted body
Pattern 3

Claim / Citation Grounding

  • 适合：research report、market analysis、technical survey、evidence-heavy deliverable
  • 机制：抽取 claims，映射 citations，验证 source support，检测 unsupported claims
  • 要求：output 必须暴露 claims 和 evidence；judge 要报告 unsupported 或 contradicted claims
  • 代表 paper：DeepResearch Bench FACT、ReportBench、MMDeepResearch-Bench TRACE/MOSAIC
Pattern 4

Rubric Decomposition With Partial Credit

  • 适合：没有单一测试能覆盖成功标准的复杂 deliverable
  • 机制：把 task 拆成 atomic criteria，逐项评分，再聚合 partial credit
  • 要求：criteria 必须 observable、falsifiable，并绑定到 output field 或 artifact
  • 代表 paper：PaperBench、MLR-Bench、DeepResearch Bench RACE
Pattern 5

LLM Judge With Human Calibration

  • 适合：难以写成 deterministic test，但可以 rubric-score 的质量判断
  • 机制：LLM judge 打 rubric 分；human-labeled subset 检查 alignment 或 disagreement
  • 要求：报告 judge model、prompt、rubric、validation set、agreement rate 和 failure examples
  • 代表 paper：DeepResearch Bench RACE/FACT、ClawBench LLM judge；HealthBench-style rubric validation 也可作为类比
Pattern 6

Expert Pairwise Preference

  • 适合：专业 deliverable，专家 taste 和 domain usefulness 很重要
  • 机制：domain experts 做 blinded pairwise comparison、ranking 和 written justification
  • 要求：成本高；reproducibility 需要 public subset、grader service 或 released pairwise labels
  • 代表 paper：GDPval

对 EvolveBench 来说，最稳妥的默认是 hybrid scoring：先 hard gates 和 objective grounding，再只对通过 grounding checks 的输出做 LLM/rubric scoring。

Scoring Contract

一个 task 只有声明了 output 怎样被检查，才算 benchmark-ready。
